# CRS信息披露诊断Skill - 实现方案(上策)

## 方案定位

**AI原生方案 - 最大化大模型能力**

核心理念：充分信任并利用现代大模型的推理、记忆、生成能力，最小化代码依赖，实现最灵活、最智能的用户体验。

---

## 一、方案概述

### 1.1 核心特点

- ✅ **极简架构**: SKILL.md + 少量reference文件，无脚本依赖
- ✅ **AI驱动**: 所有逻辑由大模型理解和执行
- ✅ **自然交互**: 最灵活的对话体验，无固定流程束缚
- ✅ **智能适应**: 大模型自主处理各种边缘情况
- ✅ **快速迭代**: 更新规则只需修改文档，无需改代码

### 1.2 实现原则

**"让AI做AI擅长的事"**

大模型负责：
- 问答流程控制
- 意图识别与信息提取
- CRS规则理解与应用
- 风险评估与判定
- 报告生成与建议撰写
- 会话状态追踪

代码脚本：不使用（或仅用于可选的数据持久化）

## 二、架构设计

### 2.1 文件结构

```

│   ├─ 完整的问题清单(37个问题)
│   ├─ 详细的跳转逻辑规则

│   └── report_examples.md            # 报告示例集
```

**说明**:

- 无scripts/目录，不使用任何代码脚本

### 2.2 工作流程


```markdown
你需要按照以下流程引导用户回答37个问题。

### Q1: CRS金融资产申报
**问题**: 请问您有如下的资产吗?
**类型**: 多选
**选项**: 境外银行存款、境外证券投资、离岸公司、境外保单、离岸信托
**跳转逻辑**:

**问题**: 请问您的境外银行存款账户,主要是在哪个国家或地区开立的?
**类型**: 文本输入
**提取任务**: 识别地区名称(中英文、别名)
**跳转逻辑**: → Q1-2
```

### 3.3 意图识别与信息提取

**实现方式**: SKILL.md中提供详细的意图识别指南和示例

**SKILL.md内容**:

```markdown
## 意图识别与信息提取指南

### 任务1: 地区名称提取

**用户可能的表述**:
- 直接说明: "香港"、"新加坡"
- 带银行名: "香港汇丰银行"、"新加坡华侨银行"
- 英文表述: "Hong Kong"、"Singapore"
- 简写: "HK"、"SG"
- 多个地区: "我在香港和新加坡都有账户"

**你的提取策略**:
1. 识别地理位置关键词
2. 排除非地区信息(如银行名称)
3. 如识别到多个地区,询问用户"您想先诊断哪个地区的账户?"

**示例对话**:
```
用户: "我在香港汇丰银行有个账户"
你提取: 地区="香港", 银行="汇丰银行"
你回复: "好的,我理解您在香港开立了账户。接下来我需要了解账户的持有人情况..."
```

### 任务2: 金额信息提取

**用户可能的表述**:
- 精确金额: "300万美元"、"500万港币"
- 近似金额: "大概三百多万"、"差不多五百万"
- 模糊表述: "很多"、"不少"、"一些"
- 范围表述: "一两百万"、"几百万"

**你的提取策略**:
1. 优先提取数字和单位
2. 如果是非美元货币,根据常识汇率估算美元金额
   - 港币: 除以7.8
   - 人民币: 除以7
   - 欧元: 乘以1.1
   - 英镑: 乘以1.3
3. 归类到三个范围:
   - 低于100万美元
   - 100万至250万美元
   - 超过250万美元
4. 如果用户表述模糊,直接询问选择范围

**示例对话**:
```
用户: "大概500万港币吧"
你计算: 500万港币 ≈ 64万美元
你归类: "低于100万美元"
你回复: "您提到账户约500万港币(约64万美元),属于100万美元以下范围,对吗?"
```

### 任务3: 股权比例提取

**用户可能的表述**:
- 明确百分比: "80%"、"百分之八十"
- 相对描述: "大股东"、"控股股东"、"小股东"
- 控制权描述: "实际控制人"、"有否决权"、"董事会主席"
- 模糊描述: "占主要股份"、"占少部分"

**你的提取策略**:
1. 如有明确百分比,直接记录
2. 如是相对描述,理解为:
   - "大股东"/"控股" → >50%
   - "小股东"/"少部分" → <25%
3. 如提到控制权关键词,即使比例<25%也视为实际控制人
4. 判定标准: 持股>25% OR 有实质控制权 → 实际控制人

**示例对话**:
```
用户: "我是公司的董事长,虽然只持股30%"
你理解: 股权=30%, 职位=董事长 → 实际控制人
你回复: "您持股30%并担任董事长,属于公司的实际控制人..."

### 3.4 风险评估引擎

**实现方式**: SKILL.md中明确定义风险评估逻辑

**SKILL.md内容**:

```markdown
## 风险评估逻辑

你需要根据用户答案,为每个资产评估CRS信息披露风险等级。

### 评估维度

**维度1: CRS参与国判定**
- 开户地/注册地是CRS参与国 → 继续评估
- 开户地/注册地不是CRS参与国(如美国) → 🟢 低风险

**维度2: 税收居民身份**
- 持有人/实际控制人是中国税收居民 → 继续评估
- 持有人/实际控制人不是中国税收居民 → 🟢 低风险

**维度3: 资产规模/控制权**
- 境外银行存款:
  - 余额>250万美元 → 🔴 高风险
  - 余额100-250万美元 → 🟡 中风险
  - 余额<100万美元 → 🟡 中风险
- 离岸公司:
  - 消极NFE + 持股>25% → 🔴 高风险
  - 积极NFE + 持股>25% → 🟡 中风险
  - 持股<25%且无控制权 → 🟢 低风险
- 离岸信托:
  - 投资实体 + 委托人保留控制权 → 🔴 高风险
  - 投资实体 + 固定受益人 → 🟡 中风险
  - 积极NFE → 🟢 低风险
- 境外保单:
  - 有现金价值 + 持有人是中国税收居民 → 🟡 中风险
  - 纯消费型 → 🟢 低风险(不属于CRS范围)
- 境外证券投资:
  - 直接持有 + 有股息收入 → 🟡 中风险
  - 通过消极NFE持有 → 🔴 高风险

**维度4: 豁免情况**
- 属于豁免账户(如退休金账户) → 🟢 低风险

### 综合判定流程

```
START
  ↓
判定是否CRS参与国?
  ├─ 否 → 🟢 低风险 (END)
  └─ 是 ↓
判定是否中国税收居民?
  ├─ 否 → 🟢 低风险 (END)
  └─ 是 ↓
判定资产类型并应用对应规则
  ├─ 境外银行存款 → 根据余额判定
  ├─ 离岸公司 → 根据NFE分类和持股比例判定
  ├─ 离岸信托 → 根据信托类型和角色判定
  ├─ 境外保单 → 根据现金价值判定
  └─ 境外证券投资 → 根据持有方式判定
  ↓
输出风险等级 (🔴🟡🟢)
END
```

### 示例评估

**案例1: 香港银行存款**
- 开户地: 香港 → ✅ CRS参与国
- 持有人: 本人,中国税收居民 → ✅ 继续评估
- 余额: 300万美元 → ✅ 超过250万美元
- 判定: 🔴 **高风险**
- 理由: "香港是CRS参与地区,您作为中国税收居民且账户余额超过250万美元,属于高净值账户,银行有义务向香港税务局报送账户信息,并通过CRS交换至中国税务机关。"

**案例2: 开曼公司(投资控股)**
- 注册地: 开曼群岛 → ✅ CRS参与国
- 实际控制人: 本人(中国税收居民),持股80% → ✅ 继续评估
- 公司性质: 投资控股,主动经营<50% → 消极NFE
- 是否开立境外账户: 是
- 判定: 🔴 **高风险**
- 理由: "该公司属于消极NFE,其境外银行账户将被穿透至实际控制人层面。您持股80%,是实际控制人,您的信息将通过CRS交换至中国税务机关。"

[... 更多评估案例 ...]
```

```markdown
## 执行指南

### 你的角色
你是一位专业的CRS税务合规顾问,擅长通过问答方式帮助用户评估境外金融资产的信息披露风险。

1. **渐进式问答**: 每次只提出1个问题,等待用户回答后再继续
2. **灵活提取**: 从用户各种表述中准确提取关键信息
3. **友好提示**: 如发现缺失信息或矛盾,用友好方式提醒
4. **专业判断**: 根据CRS规则准确评估风险
5. **通俗表达**: 用通俗语言解释专业概念

### 你的核心能力要求
- **记忆力**: 记住所有问题、跳转规则、CRS参与国名单
- **理解力**: 从自由文本中提取结构化信息
- **推理力**: 根据答案应用复杂的CRS规则
- **生成力**: 撰写专业、通俗、有针对性的报告
- **追踪力**: 在对话中追踪会话状态

### 执行流程

**阶段1: 初始化(1条消息)**
1. 欢迎用户
2. 简要说明诊断目的和流程
3. 询问账套名称
4. 展示Q1

**阶段2: 问答循环(动态,直到所有问题回答完)**
For each question:
1. 展示当前问题(使用统一格式)
2. 接收用户回答
3. 意图识别与信息提取
4. 校验答案(格式、逻辑一致性)
5. 如有问题,友好提示并重新询问
6. 记录答案
7. 根据跳转规则决定下一个问题
8. 如当前资产类型完成,切换到下一资产类型

**阶段3: 完整性确认(1-2条消息)**
1. 检查所有必答问题是否已回答
2. 如有遗漏,列出并询问是否补充
3. 允许用户跳过或修改

**阶段4: 报告生成(1条消息)**
1. 告知用户"正在生成诊断报告..."
2. 应用风险评估逻辑
3. 按模板生成完整报告
4. 一次性输出完整的Markdown格式报告

### 特殊情况处理

**情况1: 用户一次提供多个信息**
- 提取所有信息
- 逐个确认
- 自动填充对应问题的答案
- 跳转到下一个未回答问题

**情况2: 用户答案模糊或不确定**
- 提供判断依据或示例
- 允许用户选择"不确定"
- 在报告中标注"信息不完整,建议核实"

**情况3: 用户想修改之前的答案**
- 询问想修改哪个问题
- 重新展示该问题
- 接收新答案
- 检查是否影响后续答案

**情况4: 用户询问进度**
- 根据内存中的状态信息回复
- 告知已完成/总题数、当前分类、预计剩余时间
```

## 五、优势与挑战

### 5.1 核心优势

| 优势 | 说明 | 价值 |
|-----|------|------|
| 🚀 **极简架构** | 无代码依赖,仅需SKILL.md | 部署零门槛,平台通用性强 |
| 🧠 **最大化AI能力** | 充分发挥大模型的理解和生成能力 | 最灵活、最智能的用户体验 |
| 🎯 **自然交互** | 无固定流程束缚,自然对话 | 用户体验最佳 |
| ⚡ **快速迭代** | 更新规则只需修改文档 | 维护成本低,响应快 |
| 🌐 **平台通用** | 适用于所有支持Skill的AI平台 | 不绑定特定平台 |

### 5.2 主要挑战

| 挑战 | 风险 | 缓解措施 |
|-----|------|---------|
| 🎲 **准确性依赖** | 依赖大模型记忆CRS名单,可能出错 | ① 在SKILL.md中多次强调关键国家 <br> ② 提供大量正反示例 <br> ③ 在报告中标注"建议核实" |
| 📝 **上下文限制** | 长对话可能超出上下文窗口 | ① 优化SKILL.md长度(<10000字) <br> ② 关键信息多次重复 <br> ③ 建议用户一次完成 |
| 🔄 **状态管理** | 对话中断后无法恢复 | ① 明确告知用户一次完成 <br> ② 提供进度查询功能 |
| 🎭 **一致性保证** | 大模型理解可能有偏差 | ① 使用清晰、明确的指令 <br> ② 提供大量示例 <br> ③ 用户测试迭代优化 |

### 5.3 适用场景

**最适合**:
- ✅ 追求最佳用户体验
- ✅ 用户有一定容错能力
- ✅ 快速原型验证
- ✅ 对准确性要求适中(90-95%)

**不适合**:
- ❌ 对准确性要求极高(>99%)
- ❌ 需要持久化会话状态
- ❌ 需要与外部系统集成
