# 06 CRS Skill（上策）实施计划

**文档编号**：IMPL-CRS-SKILL-001
**方案来源**：方案一（上策）全链路智能对话方案
**撰写日期**：2026-03-21
**适用人员**：负责开发本 Skill 的工程师或提示词工程师

---

## 一、项目概述

本次实施目标是按照上策方案，在 Qoder Agent 平台上开发一个名为 crs-risk-assessment 的 Skill，帮助中国内地税务居民评估其海外资产在 CRS（共同申报准则）框架下是否存在信息交换风险，并生成一份专业的评估报告。

Skill 以纯对话方式运行，无需用户填写表单。核心机制有三条：每次只提一个问题的逐题引导、对每条回答执行语义意图分析、所有必填字段校验通过后才推进下一步。信息收集完毕后，触发五条判断链条逐项分析资产，最终在聊天窗口内输出完整的评估报告。

整个 Skill 由 6 个文件组成，全部存放在 `.qoder/skills/crs-risk-assessment/` 目录下。

---

## 二、文件结构与开发顺序

```
.qoder/skills/crs-risk-assessment/
├── SKILL.md                      ← 第一步：主控文件，交互规则和流程入口
├── crs-member-countries.md       ← 第二步：CRS 成员国与激活状态数据
├── questionnaire.md              ← 第三步：五类资产的分级问卷与必填字段定义
├── judgment-logic.md             ← 第四步：五条判断链条与边界规则
├── report-template.md            ← 第五步：报告四章节格式模板
└── faq-knowledge.md              ← 第六步：CRS 常见问题知识库
```

开发顺序的逻辑是：先写主控文件确立骨架，再依次补充 SKILL.md 所引用的支撑文件，最后开发知识库。每完成一个文件，在 Agent 平台上做一轮单文件验证，确认基础行为正确后再开发下一个文件，避免在多个文件同时存在问题时难以定位根因。

## 三、各文件详细开发规范

### 3.1 SKILL.md（主控文件）

这是整个 Skill 的核心，承载全部交互规则和流程控制逻辑，长度控制在 500 行以内。

**文件开头必须有如下 YAML frontmatter：**

```yaml
---
name: CRS资产信息交换风险评估
description: 该 Skill 是一个专为中国内地税务居民设计的 CRS（共同申报准则）海外资产风险评估助手。它通过全程逐题对话方式，依次收集用户持有的银行存款、上市股票、非上市股权、保险保单、信托等五类海外资产信息，对每条回答执行语义意图分析，识别遗漏和矛盾，校验必填字段完整性，最终应用五条完整判断链条生成专业评估报告。适用场景：用户提及 CRS、海外资产风险、信息交换、税务居民合规评估等关键词时触发。
---
```

**SKILL.md 正文按以下六个部分组织：**

**第一部分：角色定位**

定义智能体的角色：你是一位专业、耐心的 CRS 税务顾问助手，帮助中国内地税务居民评估海外资产的信息交换风险。语气温和、简洁、专业，不使用恐吓性语言，对用户回答始终给予肯定后再追问。

**第二部分：启动指令**

规定每次对话开始时的固定动作：首先输出开场白，说明本次评估的目的和方式（一次只聊一个问题）；紧接着提出税务居民身份确认问题，未收到明确确认前，禁止进入任何资产问题。税务居民确认后，在内部记录"已确认：中国内地税务居民"，然后按资产顺序开始提问。

开场白参考写法如下：

> "您好！我将通过几个简短的问题，帮您评估持有的海外资产在 CRS 框架下是否存在信息交换风险，并为您生成一份评估报告。我们一次只聊一个问题，您可以随时用自己的话来回答。在开始之前，请先确认一个前提：您目前是否是中国内地的税务居民（即日常居住和纳税主要在中国内地）？"

若用户确认，继续："好的，我们开始吧——请问您是否在海外持有银行存款或金融账户？"

若用户明确表示非中国内地税务居民，则说明本工具适用范围并建议咨询专业顾问，流程终止。

**第三部分：逐题引导强制规则**

以明确的约束语气写出如下规则，使用"必须/禁止/在任何情况下"等强约束词，而不是"尽量/通常"这类弱约束词：

- 在任何情况下，每次回复只允许提出一个问题。
- 每个必填字段必须作为独立问题单独提出，禁止在同一条消息中将多个字段打包询问。
- 当前问题的回答经意图分析和完整性校验通过后，方可提出下一个问题。
- 追问缺失字段时，同样遵循每次只追问一个字段的原则。

**第四部分：意图分析执行规则**

规定对用户每一条回答必须执行以下四步，并明确每步的处理逻辑：

步骤一，提取显性信息：识别用户明确说出的所有信息点，映射到当前问题对应的字段。

步骤二，提取隐性信息：识别用户话语中可以合理推断的信息（例如"通过我在开曼的信托持有"可以推断持有方式为信托、设立地为开曼）。推断信息须在下一轮回复中向用户确认后才能作为确定信息记录，禁止直接假设。

步骤三，矛盾检测：与本次对话已记录的所有信息进行比对，若发现前后矛盾（例如先说"公司没有员工"，后说"公司有固定工资支出"），礼貌指出矛盾，请用户确认。

步骤四，必填字段比对：对照 questionnaire.md 中当前问题的必填字段清单，确认是否所有字段均已获取。若有遗漏，进入追问流程；若信息已完整，记录后推进下一问题。

注意：一级问题（是/否类）只需执行步骤一和步骤四，不需要执行矛盾检测。完整的四步分析从二级问题开始执行。

**第五部分：主流程控制**

规定五类资产的收集顺序：银行存款 → 上市股票 → 非上市股权 → 保险 → 信托。

每类资产先问一级问题（是否持有），回答否则跳过，回答是则进入该资产的后续问题。每类资产的问题层级和必填字段以 questionnaire.md 为准。若用户持有多个同类资产，逐一收集，每个资产独立处理完毕后再询问是否还有其他同类资产。

中途知识问答处理：若用户提出与 CRS 相关的背景性问题，从 faq-knowledge.md 检索最相关内容，给出 2-3 句简短回答，然后用一句话引导用户回到当前问题（例如："好的，我们继续——请问[复述当前问题]？"）。若用户问题超出 CRS 范畴，简短说明评估范围并引导回正轨，不做延伸解答。

**第六部分：报告生成指令**

所有资产信息收集完毕后，执行以下动作：

对每类资产，按 judgment-logic.md 中对应的判断链条，输出评估结论（会交换 / 不会交换 / 存在风险）和自然语言段落形式的理由说明。

按 report-template.md 的四章节格式，将所有结论汇总生成完整报告。

在聊天窗口中展示报告全文，并提示用户可复制保存或截图存档；如需将报告发送给税务顾问，可手动转发。

### 3.2 crs-member-countries.md（成员国数据文件）

这个文件是"所在地是否会交换"这一判断步骤的数据来源，也是最容易出现时效性错误的文件，开发时必须核实最新数据。

**文件结构如下：**

第一部分写明数据版本日期和来源（以 OECD CRS Portal 公布的激活关系为准），并注明此文件的有效期和更新建议。

第二部分按"已与中国内地激活信息交换"和"已加入 CRS 但未与中国内地激活"两类分别列出，格式为：地区名称 + 激活状态 + 简要备注。未加入 CRS 的重要地区（如美国）单独注明。

**开发时需要核实的重点地区（以下为参考，上线前须以最新官方数据为准）：**

已激活地区（与中国内地有交换关系）包括：香港、澳门、开曼群岛、英属维尔京群岛（BVI）、新加坡、英国、德国、法国、澳大利亚、加拿大、日本、瑞士、卢森堡、列支敦士登、泽西岛、根西岛、马恩岛等主要离岸和在岸金融中心。

未加入 CRS 的地区包括：美国（采用 FATCA 体系，未加入 CRS）、巴拿马（状态需核实）、瓦努阿图（需核实激活状态）。

文件末尾写出查询使用说明：判断时对照地区名称检索此文件，若未找到对应地区，智能体应回答"该地区的 CRS 状态不在当前数据中，建议进一步核实"，并在报告中标注"存在风险"。

### 3.3 questionnaire.md（问卷与必填字段定义）

这个文件是所有问题的完整定义，包含五类资产的分级问题结构、每个问题的必填字段清单、触发条件和跳转逻辑。SKILL.md 在执行完整性校验时以此文件为依据。

**文件按五类资产分节，每节包含：**

一级问题的内容和跳转规则（回答否时跳至哪类资产）。

各级问题的完整必填字段列表，字段写法包含：字段名称、字段类型（自由输入 / 选项选择）、可选项列表（若为选项题）、是否必填。

触发条件的描述（例如：当账户类型为公司账户或信托账户时，触发三级问题）。

"不清楚"的统一处理说明：当用户对某个必填字段回答"不清楚"时，该字段记为"信息缺失"，对应资产标注为"存在风险，建议进一步分析"，并将"不清楚的字段名"记录用于报告中的理由说明。

**以银行存款为例，questionnaire.md 中的写法如下：**

一级问题：是否持有海外银行存款或金融账户（含定期、活期、股票账户、理财、基金）？必填，选项：是 / 否。回答否 → 跳至上市股票模块。

二级问题（回答是后，逐题依次询问）：

所在国家或地区，自由输入，必填，用于查询 crs-member-countries.md 的激活状态。

账户类型，选项题，选项为个人账户 / 联名账户 / 公司账户 / 信托账户，必填，选择公司账户或信托账户时触发三级问题。

账户余额大致范围，选项题，选项为 50 万人民币以下 / 50 万至 300 万 / 300 万以上，必填，用于报告展示。

三级问题（仅当账户类型为公司账户或信托账户时触发，逐题依次询问）：

该账户所属机构的主营业务类型，选项题，选项为投资理财（如基金管理、资产配置）/ 实际经营业务（如贸易、生产、服务）/ 不清楚，必填。

该机构被动收入（股息、利息、租金、资本利得）占比是否超过 50%，选项题，选项为是 / 否 / 不清楚，必填。

**保险资产的特殊处理说明：**

保险资产在一级问题后，需要立即追加一级补充问题：该保单是否具有现金价值（如终身寿险、年金险、储蓄型保险属于此类，纯消费型寿险和医疗险不属于此类）？选项为是 / 否。回答否时，说明该保单不在 CRS 交换范围，直接跳至下一类资产，后续所有字段不再收集。只有回答是时，才进入保险资产的二级问题流程。

**所有五类资产按以上结构完整定义，这是 SKILL.md 执行完整性校验时唯一的字段依据来源。**

### 3.4 judgment-logic.md（五条判断链条）

这个文件是决定最终评估结论的核心逻辑，SKILL.md 在所有信息收集完毕后触发此文件中的规则，逐项资产执行判断。

**文件结构：五条判断链条，每条链条按步骤顺序写出。**

每条链条需要包含正常路径（信息完整时的判断步骤和条件）和边界路径（用户回答"不清楚"时的处理规则、多层嵌套时的终止规则）。每条链条末尾输出格式统一为：结论（会交换 / 不会交换 / 存在风险）+ 理由（自然语言段落，说明判断依据，让用户能够真正理解，而不只是看到一个结论）。

**判断链条一：银行存款 / 金融账户**

步骤一，判断账户持有人类型。账户类型为个人或联名账户，直接跳至步骤三。账户类型为公司账户或信托账户，进入步骤二。

步骤二，判断机构性质。主营业务为投资理财，或被动收入超过 50%，认定为金融机构，穿透至实际控制人（用户），进入步骤三。主营业务为实际经营且被动收入不超过 50%，认定为非金融机构，进入步骤三。任意字段回答"不清楚"，标注为"存在风险"，记录"机构性质未确认"，跳出此链条，直接生成存在风险结论。

步骤三，查询所在地。对照 crs-member-countries.md 查询账户所在地的激活状态。已激活，结论为"会被交换"；未激活或未加入 CRS，结论为"不会被交换"。

**判断链条二：上市公司股票**

步骤一，判断持有方式。个人直接持有或通过券商账户持有，直接跳至步骤三。通过公司或信托持有，进入步骤二。

步骤二，判断持有架构的性质。架构为投资机构（被动收入超过 50%），认定为金融机构，触发穿透。架构为实际经营机构，判断是否为被动非金融机构，若是则仍需穿透，若不是则继续步骤三。任意字段回答"不清楚"，标注为"存在风险"。

步骤三，查询交易所所在地。对照 crs-member-countries.md 查询股票所在交易所的国家/地区激活状态。已激活则"会被交换"，未激活则"不会被交换"。

**判断链条三：非上市公司股权**

步骤一，判断持有方式。个人直接持有，进入步骤二。通过公司或信托持有，进入步骤三。

步骤二，判断被投资公司性质（个人直接持有时）。公司有实际经营且被动收入低于 50%，认定为积极非金融机构，结论为"不会被交换"（股权本身不在 CRS 直接交换范围）。公司为投资壳公司或被动收入超过 50%，认定为被动非金融机构，结论为"存在风险，公司账户信息可能被穿透"。

步骤三，判断上层架构性质（通过公司或信托持有时）。上层架构为金融机构或被动非金融机构，触发穿透，用户信息"会被交换"。上层架构为实际经营机构，一般不穿透，进入步骤四判断注册地。

步骤四，查询注册地。对照 crs-member-countries.md，若注册地已激活且上述判断结果为"会被交换"或"存在风险"，给出对应结论。

**判断链条四：保险（有现金价值的保单）**

步骤一，判断投保人身份。投保人为个人，直接进入步骤三。投保人为公司或信托，进入步骤二。

步骤二，判断投保人架构性质。架构为金融机构或被动非金融机构，穿透至用户，进入步骤三。

步骤三，查询保险公司所在地。对照 crs-member-countries.md，已激活则"会被交换"，未激活则"不会被交换"。

特别说明：若投保人、被保险人、受益人角色不同，在报告中明确注明"该保单信息可能被交换给投保人所在税务管辖地，建议确认投保人税务居民身份"。

**判断链条五：信托**

步骤一，判断用户在信托中的角色。五类角色的披露规则不同：委托人通常作为控制人被披露；受益人在全权信托（discretionary trust）中，所有潜在受益人都可能被披露；受托人为金融机构时，信托信息会被申报；保护人和监察人可能被认定为控制人而被披露。

步骤二，判断信托性质。信托持有金融资产且为主要业务，认定为投资机构/金融机构，信息"会被交换"。信托持有非上市公司股权，需穿透判断下层公司性质（参考判断链条三的步骤二逻辑）。

步骤三，查询信托设立地。对照 crs-member-countries.md，已激活则"会被交换"，未激活则"不会被交换"。

**通用边界规则（文件末尾统一写出）：**

当任意步骤中的信息不足无法做出判断时，该资产统一输出"存在风险，建议提供更多信息后进行专项分析"，并在报告的理由部分列明具体是哪个信息点不明确。

多层嵌套架构追问上限为第二层架构（三层关系链），超出此深度后统一标注存在风险，不再继续追问。

### 3.5 report-template.md（报告格式模板）

这个文件定义报告的四章节格式，SKILL.md 在生成最终输出时以此为格式参照。

**第一章，基本情况。**

固定文本：确认用户为中国内地税务居民，说明本报告仅针对 CRS 框架下的信息交换风险进行评估，不涵盖 FATCA 或其他合规要求。

动态内容：评估日期（当天日期）、本次评估覆盖的资产类型列表。

**第二章，资产风险分析。**

按每项资产重复以下结构：资产名称与类型 → 关键信息摘要（所在地、持有方式、架构性质等关键字段）→ 评估结论（会交换 / 不会交换 / 存在风险）→ 理由说明。

理由说明必须以自然语言段落呈现，解释清楚判断依据，包括所在地是否为 CRS 激活地区、持有方式是否触发穿透、相关角色的披露规则等。禁止只输出结论而不给理由，也禁止用机械条目列表替代段落说明。

若同一类型存在多项资产，逐项分别呈现，不合并处理。

**第三章，综合建议。**

分两类分别呈现：

"立即关注事项"列出结论为"会交换"的资产，说明这些资产的信息会被交换回中国内地税务机关，建议用户关注税务合规，必要时提前做规划。

"建议进一步核实事项"列出结论为"存在风险"的资产，具体说明每项资产需要补充哪些信息或进行什么分析，并建议用户咨询专业税务顾问。

若全部资产结论均为"不会交换"，则写明当前评估结论为无明确 CRS 交换风险，同时提示 CRS 参与国名单持续更新，建议定期重新评估。

**第四章，重要声明。**

固定文本：本报告基于用户提供信息及 CRS 通用规则生成，不构成正式税务意见；CRS 规则在各国执行中可能存在差异；资产情况发生变化时建议重新评估；如需针对特定资产进行详细规划，建议咨询专业税务顾问。

动态内容：注明本次评估所用 crs-member-countries.md 的数据版本日期，提示用户关注信息时效。

### 3.6 faq-knowledge.md（常见问题知识库）

这个文件收录 CRS 常见问题及简短解答，用于对话过程中的中途知识问答。

**文件结构采用标准问答对形式，每条包含：标准问法（一到两个常见表述）+ 简短回答（2-3 句话，只解释用户当下需要了解的内容，不展开复杂背景）。**

建议收录 20-30 条，覆盖以下几个方向：

基础概念类：CRS 是什么 / 哪些国家参与了 CRS / 中国从什么时候开始交换 / 什么信息会被交换 / 交换给谁看。

账户类型类：什么是金融机构 / 什么是被动非金融机构 / 什么是穿透规则 / 个人账户和公司账户的区别 / 50% 被动收入是怎么算的。

资产类型类：什么类型的保险在 CRS 范围内 / 信托的哪些角色会被披露 / 非上市股权什么情况会被交换 / 全权信托（discretionary trust）是什么。

处置类：如果资产被交换会怎样 / 有没有什么合法的方式不被交换（此类问题的回答只说明合法差异化处理的原则，不提供任何具体的税务规避建议，保持中立立场）。

知识库文件末尾写出检索说明：当用户提出知识性问题时，智能体在此文件中检索最相关的一条，给出对应的简短回答后，立即用一句话把用户引导回当前主流程问题。

## 四、SKILL.md 关键指令写法要点

SKILL.md 中最难写好的部分是指令的精确性：既要让大模型理解规则，又不能因为指令过于繁复导致模型选择性忽略。以下几个写法要点在开发时需要重点注意。

**强约束词的使用。** 不要写"尽量每次只问一个问题"，要写"在任何情况下，每次只允许提出一个问题"。用"必须"、"禁止"、"在任何情况下"这类强约束词，比"尽量"、"通常"效果明显更好，模型更倾向于遵守。

**意图分析的触发条件要区分清楚。** 一级问题（是/否类）只需要判断"是否有明确答案"，不需要矛盾检测，这两种情况在指令中要明确区分，避免模型在简单问题上花费过多推理资源，也避免在只有一条信息时误触发矛盾检测。

**支撑文件的引用写法要明确。** SKILL.md 中要清楚说明在什么时机读取哪个文件，例如：执行完整性校验时，参照 questionnaire.md 中当前资产模块的必填字段清单；执行判断时，读取 judgment-logic.md 中对应资产的链条；生成报告时，严格按 report-template.md 的四章节格式输出；回答知识性问题时，从 faq-knowledge.md 检索。引用文件的指令越明确，模型的行为越可预期。

**兜底规则要放在文件末尾。** 在 SKILL.md 末尾统一写出兜底规则：当遇到任何本文件未覆盖的情况时，默认回答"根据您提供的信息，该部分存在不确定性，建议后续与专业税务顾问核实"，不得自行猜测结论，不得跳过字段直接生成报告。这条规则可以防止模型在遇到未知情况时"发明"一个答案。

**报告生成指令要指定格式细节。** 仅说明"按模板生成报告"往往不够精确，需要在指令中补充：理由说明必须用自然语言段落，不得使用条目列表；多项资产要逐项分别呈现；综合建议要分"立即关注事项"和"建议进一步核实事项"两类展示。这些细节写在指令里，可以有效避免模型输出风格不一致的问题。

## 五、测试验证方案

开发完成后，通过以下八个场景验证 Skill 的正确性，每个场景独立测试，建议按顺序从简到繁进行。

**场景一：最简单路径。** 用户只持有一个香港个人银行账户。预期流程：确认税务居民身份 → 询问是否持有银行账户 → 依次收集所在地（香港）和账户类型（个人账户）→ 不触发三级问题 → 询问余额范围 → 查询香港激活状态 → 输出"会被交换"结论。重点验证：流程简洁，不出现多余追问，报告格式正确。

**场景二：多资产综合路径。** 用户持有香港个人银行账户 + 通过开曼公司持有美股 + 一个开曼全权信托（用户为受益人）。预期流程：三类资产分别收集信息，走各自的判断链条，信托部分触发角色分析和性质判断，报告包含三项资产各自独立的结论和理由说明。重点验证：多资产场景下流程不混乱，每类资产独立处理，报告结构清晰。

**场景三：意图分析提取测试。** 在询问银行账户类型时，用户回答"我在汇丰开了一个账户"。预期：智能体识别用户未明确说明账户类型，追问"请问该账户是个人账户还是公司/信托账户？"而不是直接假设为个人账户。重点验证：隐性信息不被误认为显性信息。

**场景四：矛盾检测测试。** 在非上市股权的三级问题中，用户先说"这家公司基本上没有员工"，后来又提到"公司每个月有固定的员工工资支出"。预期：智能体礼貌指出前后矛盾，请用户确认公司是否实际有员工，而不是直接以最后一条信息覆盖前一条。重点验证：矛盾检测机制正常运作，用户被引导到澄清而非被系统静默覆盖记录。

**场景五：保险现金价值筛选测试。** 用户回答持有一份纯消费型医疗保险。预期：在"是否有现金价值"问题回答否后，智能体立即说明该保单不在 CRS 交换范围，跳至下一类资产，不再询问保险公司所在地、投保人等后续字段。重点验证：筛选逻辑生效，不出现无意义的后续追问。

**场景六：多层嵌套终止规则测试。** 用户描述一个三层架构：信托持有 A 公司，A 公司持有 B 公司，B 公司持有目标资产。预期：智能体追问信托性质和 A 公司性质后，对于 B 公司及以下层级统一标注"存在风险，建议提供完整架构图后进行专项分析"，不再继续追问第三层。重点验证：追问终止边界正确执行，不出现无休止的嵌套追问。

**场景七：中途知识问答测试。** 用户在回答非上市股权问题过程中提问"什么是被动非金融机构？"。预期：智能体给出 2-3 句解释后，用一句话把用户带回当前问题，而不是在知识点上展开额外内容，也不是完全忽略用户的问题。重点验证：知识问答不打断主流程，解答简洁后立即回归正轨。

**场景八：非税务居民终止测试。** 在开场确认时，用户明确表示自己不是中国内地税务居民。预期：智能体给出说明（本工具专为中国内地税务居民设计，建议咨询专业顾问），流程终止，不进入任何资产问题。重点验证：前置确认逻辑生效，不被跳过。

## 六、开发里程碑

**第一天：** 完成 SKILL.md 初稿和 crs-member-countries.md，部署到 Agent 平台做首次单文件验证。重点验证：开场白是否正常触发、税务居民确认是否生效、基础跳转逻辑是否正确、第一类资产（银行存款）的逐题收集是否符合预期。发现问题立即修改指令写法，确认基础行为稳定后再进入第二天。

**第二天：** 完成 questionnaire.md 和 judgment-logic.md，做中期整合测试。优先覆盖场景一（最简路径）、场景二（多资产）、场景五（保险筛选），验证五类资产的问卷流程和五条判断链条能否正确执行。重点关注：必填字段是否都被收集、判断结论是否符合预期逻辑。

**第三天：** 完成 report-template.md 和 faq-knowledge.md，做完整端到端测试。覆盖全部八个测试场景，重点关注报告输出的格式和质量、知识问答的流畅度、矛盾检测和多层嵌套边界的表现。

**第四天：** 根据第三天测试结果修复问题，优化 SKILL.md 中关键指令的精准度，完成最终验收测试。验收标准：八个测试场景全部通过，报告格式符合模板要求，理由说明通俗易懂，逐题原则在所有场景下严格执行。
