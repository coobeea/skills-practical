# 03 CRS Skill 方案二（中策）：核心功能标准实现方案

**文档编号**：SOL-CRS-SKILL-002  
**方案级别**：中策  
**撰写日期**：2026-03-21  
**适用场景**：以最小必要成本覆盖核心业务需求，兼顾用户体验与开发效率

---

## 一、方案概述

本方案在坚守"逐题引导"和"意图分析"两大核心交互原则的前提下，对功能范围做出合理取舍：问题层级精简至三级（去掉部分四级追问）、知识库仅内嵌成员国列表（不含 FAQ 知识库）、报告仅支持聊天窗口展示（不含邮件发送）、不处理多层嵌套架构的深度追问。

这一方案覆盖了绝大多数用户的实际资产场景，开发工作量适中，是在资源约束下的最优平衡点。

## 二、Skill 文件结构


## 三、交互设计

### 3.1 逐题引导（与上策一致）

同样严格执行"每次只问一个问题"原则，这是本方案与上策共享的核心设计，不做妥协。

对话开场白与上策保持一致，语气亲切、简洁，让用户有清晰的流程预期。

### 3.2 意图分析（精简版）

本方案保留意图分析能力，但聚焦在最高频的两种处理场景上：

场景一——信息提取：用户的回答包含了所需信息但表达方式非结构化时，智能体自动提取并确认（如用户说"我在新加坡有个个人存款账户，大概有 100 多万人民币"，直接提取所在地=新加坡、类型=个人账户、余额=50-300 万区间）。

场景二——不完整追问：当用户回答缺少必填字段时，以友好语气追问缺失项，每次只追问一个字段。

对于矛盾检测，本方案不做主动矛盾识别（降低实现复杂度），但当用户自行纠正之前说法时，智能体应以最新说法为准。

### 3.3 中途知识问答处理

本方案不内嵌 FAQ 知识库文件。若用户提出 CRS 背景性问题，智能体依赖大语言模型自身的通用 CRS 知识进行简短回答（2 句话以内），然后引导用户回到当前问题。这一处理方式在绝大多数常见问题上足够准确，风险在于极少数专业边界问题的回答可能不够精确，但通过报告中的免责声明可以管控这一风险。

### 3.4 完整性校验（精简版）

只对每个资产类型中的核心必填字段（即直接影响判断结论的字段）做强制校验，非核心字段（如账户余额范围）若未回答则允许跳过，在报告中标注为"未提供，暂不影响主要判断"。

## 四、功能模块详细设计

### 4.1 五类资产问卷（三级结构）

**资产一：银行存款 / 金融账户**

一级：是否持有海外银行存款或金融账户？  
二级（核心必填）：所在国家或地区 / 账户类型（个人/联名/公司/信托）  
（余额范围为可选项，未提供不阻塞流程）  
三级（公司账户或信托账户时触发，核心必填）：主营业务是投资理财还是实际经营 / 被动收入是否超过 50%  

**资产二：上市公司股票**

一级：是否持有海外上市公司股票？  
二级（核心必填）：所在国家或地区 / 持有方式  
（交易所名称为可选项）  
三级（公司或信托持有时触发，核心必填）：架构主营业务 / 被动收入是否超过 50%  

**资产三：非上市公司股权**

一级：是否持有非上市公司股权？  
二级（核心必填）：公司注册地 / 持有方式  
三级（始终触发，核心必填）：公司是否有实质经营业务 / 被动收入是否超过 50%  
（持股比例为可选项；若为架构持有，追问上层架构注册地和业务性质，但不继续追问更上层）  

**资产四：保险（有现金价值的保单）**

一级：是否持有海外有现金价值的保险保单？  
二级（核心必填）：保险公司所在地 / 投保人是个人还是公司/信托  
（被保险人和受益人身份为辅助信息，若用户主动提供则记录，否则在报告中注明建议确认）  
三级（投保人为公司或信托时触发，核心必填）：架构注册地 / 是否为投资机构  

**资产五：信托**

一级：是否参与海外信托？  
二级（核心必填）：信托设立地 / 信托类型 / 用户角色  
三级（始终触发，核心必填）：信托是否属于投资机构（主要持有金融资产）  
（若信托持有非上市股权，追问被投资公司是否为实质经营公司，不做更深层追问）  

### 4.2 五条判断链条（核心规则）

判断逻辑与上策一致，区别在于边界情况处理策略：

本方案对所有无法明确判断的情况（包括多层嵌套架构、用户"不清楚"的字段）统一处理为"存在风险，建议进一步分析"，不再继续深度追问，以此换取实现简洁性。

CRS 参与国查询通过内嵌的 crs-member-countries.md 完成，判断逻辑明确、无歧义。

### 4.3 报告生成（标准格式）

遵循 report-template.md 四章节结构，内容呈现采用结构化条目（清晰、易读），不追求上策的段落叙述风格。

综合建议部分统一呈现，不区分"立即关注"和"建议核实"两个层级。

### 4.4 报告输出

仅支持聊天窗口展示，不做邮件发送功能。最后询问用户是否需要将报告内容保存时，引导用户自行复制或截图。

## 五、Skill 文件规范要求

SKILL.md 须包含：逐题引导强制指令（每次只问一个问题）、精简版意图分析要求（提取有效信息 + 追问缺失核心字段）、不推进条件（核心字段未满足时等待用户补充）、超出范围问题的简短处理方式、报告格式指令。

## 七、方案定位

本方案是"能用"与"好用"之间取得平衡的务实选择。对于资产结构不特别复杂的用户（单层持有、有明确答案的场景），体验接近上策；对于架构复杂的用户，报告中会有较多"存在风险，建议进一步分析"的条目，用户需要寻求后续专业支持。这一定位本身也是合理的——Skill 的目标不是替代专业顾问，而是帮助用户快速识别高风险点。

适用条件：资源有限但希望尽快上线；目标用户资产结构以直接持有或单层架构为主；可接受少量"存在风险"结论代替深度分析。
