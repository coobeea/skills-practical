# 02 CRS Skill 方案一（上策）：全链路智能对话方案

**文档编号**：SOL-CRS-SKILL-001  
**方案级别**：上策  
**撰写日期**：2026-03-21  
**适用场景**：对用户体验和功能完整性要求最高，愿意投入较多开发资源的场景

---

## 一、方案概述

本方案追求最高完整度的用户体验和最深度的业务覆盖。智能体在全程对话中扮演一位有耐心、有专业知识的税务顾问助手，每次只提一个问题，对用户的每一句回答都进行语义理解和意图分析，自动提取有效信息、识别矛盾和遗漏，并以温和的方式引导用户补充。知识库完整内嵌，用户在对话过程中随时可以提问，智能体能在不打断主流程的前提下进行简短解答后重回正轨。最终生成专业格式报告，支持邮件发送和聊天窗口双渠道输出。

## 三、交互设计（核心设计点）

### 3.1 严格逐题引导

全程强制执行"每次只问一个问题"原则，无论当前收集的资产有多少个信息点需要获取，都逐一提出。每问完一个问题，等待用户回答，通过校验后再继续下一个。

对话开场白统一为：
> "您好！我将通过几个简短的问题，帮您评估持有的海外资产在 CRS 框架下是否存在信息交换风险，并为您生成一份评估报告。我们一次只聊一个问题，您可以随时用自己的话来回答。"
> "在开始之前，请先确认一个前提：您目前是否是**中国内地的税务居民**（即日常居住和纳税主要在中国内地）？"

若用户确认是中国内地税务居民，继续：
> "好的，我们开始吧——请问您是否在海外持有银行存款或金融账户？"

若用户明确表示非中国内地税务居民，则说明：
> "本工具专为中国内地税务居民设计，针对其他税务居民身份的评估逻辑会有所不同。建议您咨询专业顾问，了解您所在税务管辖地的具体 CRS 申报要求。"

税务居民身份确认是整个评估流程的必要前提，在未确认前不得推进后续问题。

### 3.2 全语义意图分析

对用户的每一条回答，智能体先执行语义理解，完成以下四步判断：

第一步，提取显性信息：用户明确说出的信息点（如"香港的账户""个人账户"）。

第二步，提取隐性信息：用户话语中间接表达的信息（如"通过我在开曼设立的家族信托来持有"→ 可推断持有方式为"通过信托持有"、架构设立地为"开曼"）。

第三步，检测矛盾：对比用户本次回答与之前回答中的逻辑一致性（如之前说"公司没有实质经营"，本次又说"有固定员工"，则提示矛盾）。

第四步，比对必填字段：将已提取信息与当前步骤必填字段清单比对，识别仍缺失的字段。

### 3.3 三类处理路径

根据意图分析结果，智能体走不同处理路径：

信息完整，无矛盾 → 直接记录并推进下一步。

信息不完整 → 先肯定已提供的部分，再追问缺失字段（仍遵循每次只追问一个字段的原则）。

存在矛盾 → 礼貌指出矛盾，请用户确认，例如："您提到该公司没有实际业务，但同时提到有员工和办公室，这两点似乎有些不一致，请问该公司平时有日常的经营活动吗？"

### 3.4 中途知识问答

若用户在回答过程中提出与 CRS 相关的背景性问题（如"什么是被动非金融机构？""CRS 会把什么信息交换给中国税务局？"），智能体从 faq-knowledge.md 中检索相关内容，给出简短回答（2-3 句），然后用一句话将用户带回当前问题。

若用户的问题完全超出 CRS 范畴（如 FATCA、BEPS 等），简短说明本次评估仅覆盖 CRS，并建议专项咨询后继续引导用户回到正题。

### 3.5 完整性校验规则

每类资产的每一步骤均有明确的必填字段清单（详见 questionnaire.md），所有必填字段确认完整后方可推进。

## 四、功能模块详细设计

### 4.1 五类资产问卷（完整四级结构）

**资产一：银行存款 / 金融账户**

一级：是否持有海外银行存款或金融账户？  
二级（必填字段）：所在国家或地区 / 账户类型（个人/联名/公司/信托）/ 账户余额大致范围  
三级（公司账户或信托账户时触发，必填字段）：账户所属机构的主营业务类型 / 被动收入占比是否超过 50%  
（若持有多个账户，逐一收集，每个账户独立处理）

**资产二：上市公司股票**

一级：是否持有海外上市公司股票？  
二级（必填字段）：股票所在交易所 / 所在国家或地区 / 持有方式（个人直接/券商账户/公司或信托）  
三级（公司或信托持有时触发，必填字段）：持有架构的主营业务类型 / 被动收入占比是否超过 50%  

**资产三：非上市公司股权**

一级：是否持有非上市公司股权？  
二级（必填字段）：公司注册地 / 持有方式 / 持股比例  
三级（始终触发，必填字段）：公司主要业务描述 / 被动收入大致占比 / 是否有员工和实质经营活动  
四级（架构持有时触发，必填字段）：上层架构注册地 / 上层架构主营业务 / 上层架构被动收入占比  

**资产四：保险（有现金价值的保单）**

一级（初步确认）：是否持有海外保险保单？  
一级补充（现金价值筛选门槛，回答"是"后立即触发）：该保单是否具有现金价值（如终身寿险、年金险、储蓄型保险属于此类；纯消费型寿险、医疗险不属于此类）？若用户回答"否（无现金价值）"，则说明该保单不在 CRS 交换范围，跳至下一类资产，不再追问后续字段。  
二级（通过筛选后，按逐题原则依次收集，必填字段）：保险公司所在地 → 保单类型 → 投保人身份 → 被保险人身份 → 受益人身份 → 以上角色是否有公司或信托参与  
三级（涉及公司或信托时触发，必填字段）：架构注册地 → 主营业务类型 → 被动收入占比是否超过 50%  

**资产五：信托**

一级：是否参与设立或作为成员参与海外信托？  
二级（必填字段）：信托设立地 / 信托类型（可撤销/不可撤销/全权信托）/ 用户在信托中的角色（可多选）  
三级（始终触发，必填字段）：信托主要持有的资产类型 / 信托是否属于投资机构  
四级（信托持有非上市公司股权时触发，必填字段）：被投资公司性质（实际经营公司 vs 投资壳公司）  

### 4.2 五条判断链条（含完整边界情况）

详细判断逻辑存放于 judgment-logic.md，此处说明边界情况处理规则：

当用户对某字段回答"不清楚"时，该资产标注为"存在风险，建议进一步分析"，在报告中列明具体是哪个信息点不明确。

当一项资产涉及多层嵌套架构时（如"信托持有公司，公司再持有另一公司"），智能体最多追问至**第二层架构**（即：用户 → 第一层架构 → 第二层架构，共三层关系链）。超过此深度后，统一标注为"存在风险，建议提供完整架构图后进行专项分析"，不再继续追问。设置此终止条件的目的有两个：一是保护对话体验，避免用户陷入无休止的追问疲劳；二是控制上下文长度，防止因对话轮次过多导致模型对早期信息的推理出现偏差。

当被动收入占比用户无法精确给出时，智能体询问其是否超过一半，并以此作为判断依据。

### 4.3 报告生成（专业格式）

报告遵循 report-template.md 的四章节结构，在内容上有以下扩展：

资产风险分析章节中，对每项资产的理由说明使用自然语言段落，而非简单的条目列表，让用户能够真正理解判断逻辑。

综合建议章节中，区分"立即关注事项"（会交换的资产）和"建议进一步核实事项"（存在风险的资产），分开呈现。

报告末尾附上本次评估涉及的 CRS 参与国确认时间（对应知识库版本），提示用户关注信息时效。

### 4.4 报告发送

报告统一在**聊天窗口内展示完整文本**。由于本 Skill 运行在 Agent 平台原生环境中，不调用外部 API，因此不支持系统自动发送邮件。报告生成后，智能体引导用户将内容复制保存或截图存档，并提示如需将报告发送给税务顾问，可手动转发。

注：邮件自动发送功能依赖外部邮件服务接口，与平台技术约束冲突，在本方案中明确排除，避免给开发阶段留下误导性预期。

## 五、Skill 文件规范要求

SKILL.md 须在 description 字段中包含：触发词（CRS、海外资产风险、信息交换评估、税务居民合规）、核心能力描述（逐题对话引导、全语义意图分析、五资产判断引擎、专业报告生成）、适用对象（中国内地税务居民）。

SKILL.md 正文须包含以下强制指令：禁止一次性列出多个问题；对每一条用户回答执行四步意图分析；必填字段未满足时不得推进下一问题；中途知识问答不打断主流程；报告严格按模板格式生成。

## 七、方案优势与适用条件

本方案最大限度地还原了真实顾问与客户沟通的体验，用户无需了解 CRS 专业知识，完全依赖自然语言表达即可完成评估。报告质量高，业务覆盖完整，边界情况处理细致。

适用条件：平台已验证大语言模型在长上下文多轮对话中能稳定遵循 SKILL.md 的交互指令；愿意投入足够的开发和测试资源；目标用户资产情况较为复杂（多类资产、多层架构）。
