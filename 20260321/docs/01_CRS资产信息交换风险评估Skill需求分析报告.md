## 一、背景与目标

### 1.1 背景

CRS（Common Reporting Standard，共同申报准则）是由 OECD 主导的全球金融账户信息自动交换框架，目前已有 100+ 国家和地区加入。中国内地税务居民若在 CRS 成员国/地区持有金融资产，相关信息将被当地金融机构申报并交换回中国税务机关。

评估哪些资产会被交换、是否存在税务合规风险，是一项专业且繁琐的工作，需要结合资产类型、持有方式、架构性质、所在地等多个维度进行逐项判断。这类判断通常需要依赖税务专业顾问，普通用户难以独立完成。

### 1.2 目标

开发一个运行在 Agent 智能体平台上的标准 Skill，使智能体能够通过自然语言对话，逐步引导用户提供资产信息，并按照标准化判断逻辑对每项资产进行分析，最终自动生成《CRS 资产信息交换风险评估报告》。

### 1.3 运行平台与技术约束

本 Skill 运行在 Agent 智能体平台上，具有以下技术边界约束，在设计和实现时必须严格遵守：

不依赖任何外部 API 接口，所有判断逻辑在对话过程中由智能体自主完成。不引入额外的 NLP 处理库或第三方分析工具，意图识别和语义理解完全依赖大语言模型本身的能力。所有业务判断规则以文档形式内嵌于 Skill 中，智能体读取后即可执行，无需运行时动态调用外部资源。需要数据处理或格式化的环节，可以通过 Skill 内置脚本完成，但脚本仅用于结构化和格式化，不依赖外部服务。

## 二、用户画像

### 2.1 基本特征

目标用户为中国内地税务居民，持有一项或多项海外金融资产，资产类型涵盖银行存款、上市公司股票、非上市公司股权、有现金价值的保险保单、海外信托。资产持有方式较为多样，可能通过个人直接持有，也可能通过公司、信托等复杂架构间接持有。

### 2.2 用户行为特征

用户通常不具备 CRS 专业知识，习惯用自然语言、口语化方式表达，而非按照结构化表单逐项填写。在实际对话中，用户的回答往往存在以下几种典型情况：

回答发散，例如在被问及"账户所在地"时，顺带描述了账户余额、开户背景等无关信息。回答不完整，例如某个问题需要收集三个信息点（所在地、账户类型、余额范围），用户只提到了其中一到两个。回答模糊，例如"好像是在香港还是新加坡，记不太清楚"，无法直接用于判断。回答超出范围，例如用户主动询问 CRS 政策背景或其他税务问题，而非回答当前问题。

这些特征要求 Skill 在交互设计上必须具备足够的引导能力、意图识别能力和容错能力，而非简单地按照固定脚本推进。

## 三、交互设计需求

这是本 Skill 最核心的设计区域，直接决定用户体验质量。

### 3.1 逐题引导原则（强制要求）

智能体在整个信息收集过程中，必须严格遵守"每次只问一个问题"的原则。禁止在同一条消息中一次性列出某个资产类型下的所有问题，无论该资产有多少个问题需要收集。

正确的做法是：当前问题得到有效回答并通过完整性校验后，才进入下一个问题。如果当前问题包含多个信息点（如二级问题中需同时收集所在地、账户类型、余额范围），则先发出问题，等待用户回答，再校验是否完整，不完整则追问补充，完整后才推进下一步。

这一原则的目的是避免用户面对"表单式"的信息收集体验，保持对话的自然节奏，让用户感受到是在和一个有耐心的顾问沟通，而不是在填写复杂的问卷。

示例说明——错误做法：
> "请告诉我：①账户所在地 ②账户类型（个人/公司/信托）③账户余额范围 ④该账户主营业务 ⑤被动收入占比"

示例说明——正确做法：
> "好的，先了解一下这个账户的基本情况：请问这个账户开在哪个国家或地区？"
> （用户回答后）"好的，这个账户的类型是个人账户、联名账户、公司账户还是信托账户？"
> （用户回答后，若为公司账户，继续追问）"了解，那这家公司的主要业务是什么？是投资理财类，还是有实际经营业务？"

### 3.2 问题的逻辑层级结构

五类资产的问题均采用分级触发结构，每一级问题的展示依赖上一级的回答结果。具体结构如下：

**一级问题**：确认用户是否持有该类资产。若回答"否"，直接跳至下一类资产，不再展示后续问题。

**二级问题**：收集该类资产的基础信息（所在地、账户类型、持有方式等）。二级问题中往往包含多个信息点，每个信息点独立提问，不一次性打包。

**三级问题**：根据二级问题的回答有条件触发。例如账户类型为"公司账户"时，才触发关于公司主营业务和被动收入占比的追问。

**四级问题**：根据三级问题的特定回答有条件触发，通常用于收集上层架构（持有公司或信托）的详细信息。仅在非上市公司股权和信托两类资产中出现。

各类资产的完整问题层级映射关系，详见 questionnaire.md 文件。

### 3.3 用户意图分析与回答识别

由于用户的表达方式千变万化，智能体在接收到用户回答后，必须先进行意图分析，再决定下一步动作。意图分析不依赖外部 NLP 工具，而是利用大语言模型的语义理解能力，在 Skill 指令中明确要求智能体执行以下判断：

第一，判断用户的回答是否与当前问题相关。如果用户的回答明显偏离当前问题（例如当前问题询问账户所在地，但用户开始描述账户的开户经历），智能体应友好地将用户引回当前问题，例如："感谢您的介绍，我们先聚焦在账户所在地这个问题上——请问这个账户开在哪个国家或地区？"

第二，判断用户回答中是否包含了当前问题所需的关键信息。有时用户的表达是隐性的，例如当被问及"持有方式"时，用户回答"我通过我在香港设立的家族信托来持有这些股票"，智能体应识别出这句话已经包含了"通过信托持有"这一关键信息，无需再次追问。

第三，当用户回答涉及多个信息点，但只覆盖了其中一部分时，智能体应在确认已获取的部分之后，针对缺失的信息点进行友好补充追问，而不是简单重复整个问题。

第四，当用户的回答存在矛盾或逻辑不一致时（例如说"没有实质经营"但又说"有员工和办公室"），智能体应礼貌地指出矛盾并请用户确认，例如："您提到该公司没有实质经营业务，但同时提到有固定员工和办公室，这两点似乎有些不一致，请问该公司实际上是否有日常经营活动？"

### 3.4 答案完整性校验机制

每个问题步骤都有其对应的"必填信息点"清单。在推进到下一步之前，智能体必须确认当前步骤的所有必填信息点已全部获得有效回答。

各步骤必填信息点定义如下：

**银行存款 / 金融账户**  
二级问题必填：所在国家或地区、账户类型（个人/联名/公司/信托）、余额范围。三级问题（公司/信托账户时）必填：主营业务类型、被动收入占比是否超过 50%。

**上市公司股票**  
二级问题必填：交易所所在地、持有方式。三级问题（公司/信托持有时）必填：架构主营业务、被动收入占比是否超过 50%。

**非上市公司股权**  
二级问题必填：注册地、持有方式、持股比例。三级问题必填：公司主要业务描述、被动收入大致占比、是否有实质经营。四级问题（架构持有时）必填：上层架构注册地、主营业务、被动收入占比。

**保险**  
二级问题必填：保险公司所在地、保单类型、是否有现金价值、投保人身份、被保险人身份、受益人身份、是否有公司/信托参与。三级问题（涉及公司/信托时）必填：架构注册地、主营业务、被动收入占比。

**信托**  
二级问题必填：设立地、信托类型、用户在信托中的角色。三级问题必填：信托持有资产类型、是否为投资机构。若信托持有非上市公司股权，还需必填：被投资公司性质（实际经营 vs 投资壳）。

当检测到必填信息点未完整回答时，智能体需要以温和、明确的语气提醒用户补充，例如："我注意到还有一个信息没有确认到——请问这个账户的类型是个人账户还是公司/信托账户呢？"

### 3.5 友好提醒的语气规范

所有提醒和追问必须保持友好、简洁、无压迫感的语气。具体要求：

提醒时先肯定用户已提供的信息，再指出缺失部分，例如"您提到了账户在香港，还需要确认一下账户类型……"。不使用带有负面含义的措辞，例如"您没有回答"、"信息不完整"，改用"还有一个信息需要确认"、"请补充一下"。每次追问只针对一个缺失信息点，不一次性列出所有缺失项（与逐题引导原则保持一致）。

## 四、功能需求

### 4.1 对话式信息收集

智能体按照五大资产类型依次推进信息收集，整体顺序固定（银行存款 → 上市股票 → 非上市股权 → 保险 → 信托），每类资产完成后再进入下一类。在每类资产内部，严格按照分级提问结构逐步推进，遵循第三章所述的逐题引导原则和完整性校验机制。

若用户在某类资产的一级问题回答"否"（即不持有），则跳过该类资产的所有后续问题，直接进入下一类资产的一级问题。

若用户同时持有多个同类资产（例如在多个国家各有一个银行账户），则对每一个分别进行信息收集，每一个账户作为独立条目记录，分别在报告中单独分析。

### 4.2 逐项资产判断引擎

完成所有信息收集后，智能体对每一项资产按照对应的判断链条进行逐步推理。判断链条的核心逻辑如下：

**金融机构认定**：若某实体（公司或信托）的主营业务为投资理财，或被动收入占比超过 50%，则认定为"金融机构"，其账户信息直接进入 CRS 申报范围。

**被动非金融机构穿透**：若某实体主要持有资产、无实质经营（即"被动非金融机构"），则需穿透至实际控制人，实际控制人（即用户）的信息会被交换。

**积极非金融机构豁免**：若某公司有实质经营且被动收入占比低于 50%，属于"积极非金融机构"，其股权不直接属于 CRS 信息交换范围。

**CRS 参与地确认**：资产所在地（账户开户地、股票交易所所在地、公司注册地、保险公司所在地、信托设立地）是否为与中国内地激活了交换关系的 CRS 参与国/地区，是最终决定是否交换的关键条件。

**信托角色披露规则**：委托人通常作为"控制人"被披露；全权信托（Discretionary Trust）的所有潜在受益人均可能被披露，即使尚未实际获得分配；保护人/监察人可能被认定为控制人而被披露；受托人若为金融机构，信托信息将被申报。

每项资产的判断结论为以下三种之一：会交换、不会交换、存在风险（建议进一步分析）。当用户有"不清楚"的回答时，该项资产统一标注为"存在风险"。

### 4.3 综合报告生成

完成所有资产收集与判断后，智能体按照标准报告模板生成完整的《CRS 资产信息交换风险评估报告》，报告结构严格遵照以下四个部分：

**一、基本情况**：记录报告编号、评估日期、客户名称，确认客户为中国内地税务居民，说明报告评估范围仅针对 CRS 框架下的信息交换风险。

**二、资产风险分析**：逐项列出每一项资产的所在地、持有方式、其他关键信息（根据资产类型不同而不同，例如公司业务性质、保单角色、信托角色等）、评估结论，以及用通俗语言写成的判断理由。每一项资产单独占一个区块，格式统一。

**三、综合建议**：汇总"会交换"资产列表，汇总"存在风险"资产列表并给出针对性建议（如梳理公司业务实质、提供完整架构图、确认信托角色等），若全部无风险也明确告知，最后建议用户如需深入规划可安排专项咨询。

**四、重要声明**：声明报告基于用户提供的信息及 CRS 通用规则生成，不构成正式税务意见，CRS 规则在各国执行中可能存在差异，如资产情况发生变化建议咨询专业税务顾问。

### 4.4 报告发送方式

所有资产信息收集完毕后，智能体询问用户希望如何接收报告：发送至邮箱，或直接在聊天窗口中展示。根据用户选择执行对应动作。

## 五、非功能需求

**语言风格**：全程使用通俗易懂的中文表达，将 CRS 专业术语转化为普通人能理解的表达方式。在首次出现专业概念时，用括号或简短补充进行解释，例如"被动收入（包括股息、利息、租金、资本利得等）"。

**交互节奏**：问题按资产类型分块，每类资产之间有自然的过渡语，例如"好的，银行账户这部分已经了解清楚了，接下来我们来看看您是否持有海外上市公司的股票……"。

**条件跳转明确**：对于不持有某类资产的用户，跳过后有简短确认，例如"好的，这类资产我们跳过"，而不是静默推进，让用户知道流程在正常推进。

**知识库集成**：Skill 内嵌经整理的 CRS 成员国及激活交换关系列表，智能体在判断资产所在地时直接查阅该列表，无需调用外部接口。

**容错兜底**：当用户表达极度模糊或完全无法判断时，智能体不应强行得出结论，而是在报告中如实记录该信息的不确定性，并注明"建议确认后补充"。

## 六、执行流程详细说明

### 6.1 第一阶段：信息收集与结构化

本阶段是整个 Skill 执行时间最长的阶段，涵盖五类资产的全部问题交互。完整流程如下：

智能体首先简短介绍本次评估的目的和大致流程，让用户有心理预期，例如："我将通过几个简单的问题，了解您持有的海外资产情况，最后为您生成一份 CRS 信息交换风险评估报告。我们一起来逐步完成，每次只需要回答一个问题就好。"

随后依次对五类资产进行信息收集，每一类资产内部严格遵守逐题引导原则和完整性校验机制（见第三章）。所有问题收集完毕后，智能体将用户提供的信息整理成结构化数据存储，作为第二阶段判断的输入。

### 6.2 第二阶段：逐项资产分析

本阶段智能体内部执行判断，用户可能不感知，也可能需要针对"不清楚"的回答做最后确认。

对于每一项资产，智能体按照对应的判断链条（五类资产对应五条判断链条）依次执行：先判断持有方式，再判断架构性质（是否需要穿透），再查询所在地是否为 CRS 参与国，最终综合得出结论（会交换 / 不会交换 / 存在风险）并生成通俗化的理由说明。

判断优先级规则：穿透判断优先于所在地判断，即若架构判断结论为"需要穿透"，则即使所在地不在 CRS 参与国，也不能直接得出"不会交换"的结论，仍需标注"存在风险"。

### 6.3 第三阶段：综合分析与报告生成

汇总所有资产的判断结论，分为三类分别处理：明确会交换的资产、存在风险需进一步分析的资产、明确不会交换的资产。生成综合建议后，按照标准报告模板填充所有内容，完成报告，按用户选择的方式发送或展示。

## 七、技术实现评估

本章对各功能模块的实现方式进行评估，明确哪些依赖规则逻辑可以直接完成，哪些需要借助脚本处理，哪些由大语言模型自身能力承载。本 Skill 的整体原则是最大化利用大语言模型的内置能力，规则内嵌于文档，脚本仅用于必要的数据处理。

### 7.1 规则驱动可直接实现的部分

以下功能通过 Skill 文档中内嵌的规则描述，由智能体阅读后直接执行，无需额外脚本或外部服务：

**条件跳转逻辑**：一级问题回答"否"时跳至下一类资产，三级问题的触发条件判断（如账户类型为公司账户才触发三级问题）。这些都是简单的 if/else 逻辑，通过 Skill 指令中的明确描述即可让模型正确执行。

**被动收入阈值判断**：50% 阈值的比较判断，属于明确的规则，模型可直接执行。

**金融机构 / 被动非金融机构 / 积极非金融机构的分类**：基于主营业务性质和被动收入占比的分类规则，通过文档清晰描述判断树即可。

**CRS 参与地查询**：通过内嵌的 CRS 成员国知识库文件，模型读取后可直接进行匹配判断，无需调用外部 API。

**信托角色与披露规则对应**：委托人、受益人、保护人、受托人对应不同的披露义务，属于固定规则，通过文档描述清楚即可。

**判断结论的三分类**（会交换 / 不会交换 / 存在风险）：判断树终点的结论输出，规则明确，模型直接执行。

### 7.2 依赖大语言模型能力承载的部分

以下功能不需要规则或脚本，完全依赖大语言模型的内置理解和生成能力：

**意图分析与语义识别**：理解用户发散性、隐性、口语化的表达，从中提取结构化信息。这是大语言模型的核心能力，无需额外实现，只需在 Skill 指令中明确要求模型执行此项任务即可。

**回答完整性校验**：将用户的自然语言回答与必填信息点清单进行比对，判断哪些信息已获取、哪些仍缺失。这是语义理解任务，模型可直接完成。

**友好提醒文本生成**：根据缺失信息点生成语气温和、简洁的补充追问。属于文本生成任务，模型自主完成，无需模板脚本。

**通俗化判断理由生成**：将判断链条的推理过程转化为用户易理解的语言表达，属于语言转换任务，模型直接完成。

**对话上下文管理**：跨多轮对话记住已获取的信息，在后续问题中避免重复询问。依赖大语言模型的上下文窗口能力。

### 7.3 需要脚本辅助的部分

以下功能需要通过 Skill 内置脚本处理，以保证输出的格式规范性和一致性：

**结构化数据整理**：将多轮对话中收集到的用户信息，按照五类资产的数据结构进行整理和归并，确保同类资产的多个条目被正确组织。可通过一个简单的 Python 脚本或 Skill 内置的数据整理指令完成。

**报告文档格式化组装**：按照报告模板，将判断结论和理由填入对应位置，生成格式统一的报告文本。这部分建议通过报告模板文件（report-template.md）配合模型的填充能力实现，不需要独立脚本。

### 7.4 技术实现总结

综合评估，本 Skill 的技术实现方案具有较低的外部依赖风险，核心逻辑均可在 Agent 平台原生能力范围内完成。实现难点主要集中在两个方面：第一是问题逻辑层级和触发条件的描述要足够精确，确保模型按照设计意图执行；第二是意图分析和完整性校验的 Skill 指令设计，需要通过充分的提示词工程（Prompt Engineering）在 SKILL.md 中清晰表达。

## 八、Skill 设计要求

### 8.1 Skill 基本信息

名称：crs-risk-assessment  
存储位置：项目级（.qoder/skills/crs-risk-assessment/）  
触发场景：用户提及 CRS、海外资产信息交换、税务居民合规评估、CRS 风险排查等相关话题时自动触发。

### 8.2 文件结构

```
crs-risk-assessment/
├── SKILL.md                    # 主文件：Skill 描述、交互规范、工作流程入口
├── questionnaire.md            # 五类资产的完整分级问题清单（含必填信息点定义）
├── judgment-logic.md           # 五条判断链条的详细逻辑规则
├── report-template.md          # 标准报告模板
└── crs-member-countries.md     # CRS 成员国及与中国内地激活交换关系清单
```

### 8.3 SKILL.md 核心内容要求

SKILL.md 的 description 字段须包含：触发条件（CRS 评估、海外资产风险排查）、核心能力（逐题引导问卷、意图分析、完整性校验、判断引擎、报告生成）、适用对象（中国内地税务居民）。

SKILL.md 的正文内容须包含以下关键指令：

逐题引导强制规范（每次只问一个问题，禁止一次性列出多问题）；用户意图分析要求（识别语义、提取隐性信息、发现矛盾）；完整性校验要求（每步推进前确认必填信息点已全部获取）；友好提醒语气规范；对各支撑文件的引用入口。

SKILL.md 正文控制在 500 行以内，详细的问题清单、判断逻辑、报告模板分别放在对应的支撑文件中，通过引用的方式在需要时读取。

### 8.4 判断逻辑关键规则（必须在 judgment-logic.md 中明确）

被动收入占比阈值为 50%，超过则认定为金融机构或被动非金融机构。穿透判断优先于所在地判断，不得因所在地不在 CRS 参与国而跳过穿透分析。信托全权受益人（Discretionary Beneficiary）即使尚未实际获得分配，也应在报告中列为潜在披露对象。用户回答"不清楚"的项目，统一处理为"存在风险，建议进一步分析"，在报告中注明缺失信息类型。多个同类资产各自独立分析，不可合并处理。

## 九、知识库依赖说明

Skill 运行依赖以下知识库内容，需在开发阶段整理并以文档形式内嵌：

**CRS 成员国与激活状态列表**：来源文档《CRS成员国2023年.md》，需整理为"国家/地区名称 → 是否为CRS成员 → 是否与中国内地激活双向交换"的结构化清单，供智能体在判断所在地时直接查阅。

**CRS 常见问答知识库**：来源文档《CRS-related Frequently Asked Questions》和《国内版 CRS 常见问题.md》，用于处理用户在对话过程中提出的背景性问题，智能体可在不打断主流程的前提下简短回答，然后引导用户回到当前问题。

**监管法规背景**：来源文档《国家税务总局公告2017年第14号》和《中国人民银行 国家税务总局 国家外汇管理局》相关规定，作为判断逻辑的法律依据背景，不直接在对话中引用，但在报告声明中作为规则来源背书。

## 十、开发优先级与阶段建议

**第一优先级（MVP 核心功能）**：逐题引导交互逻辑、五类资产分级问卷（含触发条件）、意图分析与完整性校验规范、五条判断链条、标准报告模板生成。这是 Skill 最小可用版本，必须在首个版本中完整实现。

**第二优先级（体验完善）**：CRS 成员国知识库内嵌（实现所在地自动判断）、"不清楚"容错处理规范化、多资产条目的独立收集与汇总。

**第三优先级（高级功能）**：报告邮件发送功能、对话中途的 CRS 背景知识问答支持、历史对话记录管理。

## 十一、风险与约束

**信息准确性风险**：CRS 各国执行细则存在差异，Skill 生成的报告基于通用规则，必须在报告重要声明中明确告知用户不构成正式税务意见。

**知识库时效性风险**：CRS 成员国列表及各国与中国内地的激活状态可能随时间变化，crs-member-countries.md 文件需定期核实更新。

**模型执行一致性风险**：逐题引导、意图分析、完整性校验等核心交互行为依赖大语言模型的指令遵从能力，需要在 SKILL.md 中通过清晰、强制性的指令描述来提高一致性，并在开发完成后进行充分的场景测试。

**问题设计边界**：当前问卷精简聚焦于 CRS 信息交换判断，不涵盖 FATCA、BEPS 反避税规则等其他税务合规领域。当用户提出超出范围的问题时，智能体应简短说明边界并建议咨询专业顾问，不尝试作出超出设计范围的判断。
