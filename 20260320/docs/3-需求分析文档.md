# CRS信息披露诊断技能 - 需求分析文档

## 一、项目背景与目标

### 1.1 项目背景

CRS（共同申报准则，Common Reporting Standard）是OECD推动的全球税务透明化机制。中国已与多个国家和地区签署税收情报自动交换协议，境外金融资产信息将自动交换至中国税务机关。

许多高净值人士拥有境外金融资产（银行存款、证券投资、离岸公司、境外保单、离岸信托等），但对CRS信息披露规则了解有限，难以准确评估自身面临的信息披露风险。

### 1.2 项目目标

创建一个Qoder平台的Agent Skill（技能），通过引导式问答收集用户境外资产信息，基于CRS规则进行风险评估，生成诊断报告。

### 1.3 适用人群

拥有以下一种或多种境外资产的中国税收居民：境外银行存款账户、境外证券投资账户、离岸公司（BVI、开曼等）、境外保单（储蓄型、年金型）、离岸信托（家族信托等）。

## 二、Skill定位与设计理念

### 2.1 什么是Skill

Skill是Qoder平台的技能文件，本质上是一个**提示词工程产物**，而非代码程序。

Skill的核心是SKILL.md文件，通过markdown格式的指令和知识，指导大模型完成特定任务。大模型本身具备理解、推理、生成的能力，Skill只需要提供大模型不具备的**领域知识**和**任务指令**。

### 2.2 Skill的能力边界

**大模型本身能做的（不需要在Skill中详细定义）**

- 理解用户输入的自然语言
- 判断用户回答是否与问题相关
- 识别用户的操作意图（跳过、返回、退出等）
- 生成友好的对话回复
- 根据规则进行逻辑推理和判断
- 按照模板格式生成报告

**Skill需要提供的（大模型不知道的领域知识）**

- CRS的完整问题清单及跳转逻辑
- CRS参与国名单
- 风险评估的判定规则
- 实体分类的标准（积极NFE/消极NFE等）
- 报告的输出模板
- 专业术语的解释

### 2.3 Skill文件结构

Skill采用目录结构，包含主文件和辅助文件：

**主文件 SKILL.md**

包含YAML元数据（name、description）和核心指令。控制在500行以内，保持精简。

**辅助文件（按需引用）**

- questions.md：完整问题清单
- rules.md：评估规则和判定逻辑
- templates.md：报告模板

大模型在需要时读取辅助文件获取详细信息，实现"渐进式披露"。

### 2.4 Skill的触发机制

通过SKILL.md的description字段定义触发场景。当用户输入匹配以下关键词时，自动激活此Skill：CRS、境外资产、税务申报、信息披露、海外账户、离岸公司、境外保单、离岸信托等。

## 三、核心功能设计

### 3.1 工作流程概述

Skill的工作流程分为三个阶段：

**阶段一：引导式问答**

通过逐个问题引导用户回答，收集境外资产信息。问题根据用户回答动态跳转。

**阶段二：风险评估**

基于收集的信息，运用CRS规则判定各类资产的信息披露风险。

**阶段三：报告生成**

按照模板格式生成诊断报告，包含判定结果和合规建议。

### 3.2 大模型需要遵循的核心原则

**原则一：逐题引导**

每次只展示一个问题，等用户回答后再继续下一个。不能一次给出多个问题。

**原则二：答案完整性检查**

对于包含多个要素的问题，确保用户回答覆盖所有要素，否则进行追问。

**原则三：灵活应对**

用户可能回答无关内容、询问概念、要求跳过等，需要灵活处理并引导回正轨

## 四、问题清单分析

### 4.1 问题分类体系

问题清单共包含37个问题，按资产类型分为6个大类：

**第1类：CRS金融资产申报（入口问题）**

- 问题数量：1个
- 核心功能：确定用户拥有的资产类型，决定后续问题分支

**第2类：境外银行存款**

- 问题数量：8个
- 问题编号：1-1 至 1-4
- 核心关注点：开户地区、持有人类型、税收居民身份、账户类型、余额范围、是否豁免账户

**第3类：境外证券投资**

- 问题数量：5个
- 问题编号：2-1-1 至 2-2
- 核心关注点：开户地区、持有人类型（含穿透实体）、税收居民身份、投资产品类型

**第4类：离岸公司**

- 问题数量：8个
- 问题编号：3-1-1 至 3-3-4
- 核心关注点：注册地区、是否金融机构、经营性质、NFE分类、股权与控制权、是否开立境外账户

**第5类：境外保单**

- 问题数量：6个
- 问题编号：4-1-1 至 4-2-3-1
- 核心关注点：保单性质（是否有现金价值）、持有人、被保险人、受益人及各方税收居民身份

**第6类：离岸信托**

- 问题数量：9个
- 问题编号：5-1-1 至 5-4-1
- 核心关注点：用户角色（委托人/受益人/保护人）、税收居民身份、信托资产类型、保留权力、信托活动类型

### 4.2 问题跳转逻辑

问题之间存在三种跳转关系：

**顺序跳转**

完成当前问题后，直接进入下一个指定问题

示例：问题1-1回答后 → 直接进入问题1-2

**条件跳转**

根据用户选择的选项，进入不同的问题分支

示例：问题1-2"持有人类型"
- 选择"本人独有" → 问题1-2-1
- 选择"与配偶/家人联名" → 问题1-2-2
- 选择"由他人名义持有" → 问题1-2-3
- 选择"不确定" → 问题1-2-3

**分类结束跳转**

当前资产分类的问题全部完成后，检查用户是否还有其他资产类型，如有则进入下一个资产分类的起始问题

### 4.3 问题类型定义

问题清单中包含三种问题类型：

**单选题**

用户只能选择一个选项，如税收居民身份、账户类型等

**多选题**

用户可以选择多个选项，如资产类型、投资产品类型、信托角色等

**文本输入题**

用户需要自由文本回答，如开户地区、联名持有人详情、股权比例说明等

### 4.4 问题间的依赖关系

**前置依赖**

某些问题只有在前序问题给出特定答案时才会出现

示例：问题4-2-1（保单持有人）仅在问题4-1-1回答"有现金价值"时才会出现

**角色分支依赖**

离岸信托部分根据用户角色（委托人/受益人/保护人）展示不同的问题分支

示例：
- 委托人角色 → 展示问题5-2-A系列
- 受益人角色 → 展示问题5-2-B系列
- 保护人角色 → 展示问题5-2-C系列
- 用户可能同时具有多个角色，需依次询问各角色相关问题

## 五、问答流程设计

Skill需要定义清晰的问答流程，指导大模型如何引导用户完成问答。

### 5.1 问答启动流程

**步骤一：欢迎与说明**

向用户介绍CRS信息披露诊断的目的和流程，说明大约需要回答的问题数量和预计耗时

**步骤二：账套创建**

询问用户希望使用的账套名称，用于保存诊断结果

建议格式：姓名或代号加年份，如"张先生_2026"

**步骤三：资产类型确认**

展示初始问题（问题1），让用户选择拥有的资产类型，这决定了后续需要询问哪些问题分支

### 5.2 单个问题的处理流程

**步骤一：问题展示**

展示当前问题的完整信息，包括问题内容、说明、提示和选项

**步骤二：等待用户回答**

等待用户输入答案

**步骤三：意图识别**

分析用户的回答，判断是否与当前问题相关

**步骤四：答案校验**

对于相关回答，检查答案是否完整、是否符合问题要求

**步骤五：结果处理**

如果回答有效，记录答案并确定下一个问题
如果回答无效，给出提示并重新询问

### 5.3 意图识别场景

**识别场景一：无关回答**

用户的回答与当前问题完全无关

示例：
- 当前问题："请问您的境外银行存款账户，主要是在哪个国家或地区开立的？"
- 用户回答："今天天气真好"

处理方式：友好提醒用户当前正在进行CRS信息披露诊断，重新展示当前问题

**识别场景二：询问其他问题**

用户在回答过程中询问了不相关的问题

示例：
- 当前问题："请问您是中国税收居民吗？"
- 用户回答："什么是税收居民？"

处理方式：简要解释相关概念，然后引导用户回到当前问题

**识别场景三：想要跳过或退出**

用户表达不想回答当前问题或想要退出诊断

示例：
- 用户回答："这个问题我不想回答"
- 用户回答："我不做了"

处理方式：对于想跳过的情况，说明该问题的重要性，询问是否确定跳过。对于想退出的情况，询问是否保存当前进度。

**识别场景四：回答模糊**

用户的回答不够明确，无法对应到具体选项

示例：
- 当前问题是单选题
- 用户回答："可能是A也可能是B"

处理方式：引导用户给出更明确的回答，或记录为"不确定"

### 5.4 答案校验规则

**校验规则一：选项有效性**

对于选择题，检查用户的回答是否匹配有效选项

**校验规则二：必填完整性**

检查用户是否回答了问题中的所有必填部分

示例：
- 问题1-2-2："请分别说明联名持有人的身份及其是否是中国的税收居民"
- 这个问题包含两个隐含问题：(1)联名持有人的身份，(2)是否是中国税收居民
- 用户回答"配偶"只回答了身份，未回答税收居民身份，需要追问

**校验规则三：格式合理性**

对于文本输入题，检查回答是否合理

示例：开户地区应该是一个地理位置名称，而不是数字或乱码

**校验规则四：多选题完整性**

对于多选题，提醒用户是否还有其他选项需要选择

### 5.5 特殊问题处理

**多角色问题处理**

离岸信托部分，用户可能同时具有多个角色（委托人、受益人、保护人）

处理流程：
1. 首先询问用户在信托中的角色（多选）
2. 依次针对每个角色询问对应的问题分支
3. 各角色问题完成后，询问通用问题

**敏感信息提示**

在涉及资产金额、股权比例等敏感信息时，提醒用户：
- 信息仅用于评估目的
- 建议使用概括性描述或脱敏处理

### 5.6 问答进度提示

**进度展示**

在每个问题开始时，展示当前进度

格式示例："境外银行存款 第3/8个问题"

**断点续答**

支持保存当前问答进度，用户可以在后续会话中继续

**回溯修改**

支持用户在问答过程中回顾和修改之前的回答

## 六、风险评估规则

Skill需要包含完整的风险评估规则，这是大模型不具备的领域知识。

### 6.1 评估维度

**维度一：CRS参与国判定**

判断资产所在国家/地区是否为CRS参与国

- CRS参与国：香港、新加坡、瑞士、英国、澳大利亚、加拿大等100多个国家和地区
- 非CRS参与国：美国（使用FATCA）
- 判定结果影响后续所有评估

**维度二：税收居民身份判定**

判断账户持有人、实际控制人、受益人等是否为中国税收居民

- 中国税收居民：境外资产信息将通过CRS交换至中国税务机关
- 非中国税收居民：中国税务机关无法通过CRS获取相关信息

**维度三：实体分类判定**

针对离岸公司和信托，判定其CRS分类

- 金融机构（FI）：需履行报送义务
- 积极非金融机构（Active NFE）：相对低风险
- 消极非金融机构（Passive NFE）：需穿透至实际控制人
- 投资实体：需穿透至委托人/受益人

**维度四：穿透规则应用**

判断是否需要穿透至实际控制人层面

- 消极NFE持有的账户需穿透
- 投资实体信托需穿透至委托人/受益人
- 判断实际控制人的税收居民身份

**维度五：账户余额/价值**

根据账户余额范围判断风险等级

- 超过250万美元：高净值账户，银行加强尽职调查
- 100万至250万美元：需完成标准尽职调查
- 低于100万美元：仍需尽职调查，但相对简化

**维度六：豁免账户判定**

判断账户是否属于CRS豁免类型

- 退休金账户、学生账户等可能豁免
- 纯消费型保险不属于金融账户

### 6.2 风险等级定义

**高风险**

存在明确的CRS信息披露义务，信息已交换或即将交换至中国税务机关

触发条件（满足任一）：
- 用户是中国税收居民 + 账户在CRS参与国 + 余额超过100万美元或涉及控制权
- 离岸公司属于消极NFE + 开立境外账户 + 控制人是中国税收居民
- 信托属于投资实体 + 开立境外账户 + 委托人/受益人是中国税收居民且保留控制权

**中风险**

可能存在CRS信息披露义务，需进一步核实或存在不确定性

触发条件：
- 用户是中国税收居民 + 账户在CRS参与国 + 余额低于100万美元或间接持有
- 信托受益人是中国税收居民，但属于酌定受益人
- 存在"不确定"类回答，无法准确判定

**低风险**

无CRS信息披露义务，或不涉及中国税务机关

触发条件：
- 账户在非CRS参与国（如美国）
- 账户持有人/实际控制人不是中国税收居民
- 属于豁免账户（退休金、纯消费型保险等）
- 离岸公司属于积极NFE

### 6.3 各资产类型评估逻辑

**境外银行存款评估逻辑**

输入参数：
- 开户地区（bank_country）
- 持有人类型（bank_holder_type）
- 是否中国税收居民（bank_is_china_resident）
- 账户类型（bank_account_type）
- 账户余额范围（bank_balance_range）
- 是否特殊账户（bank_special_type）

评估流程：
1. 判断开户地区是否为CRS参与国
2. 判断持有人的税收居民身份
3. 根据余额范围确定风险等级
4. 检查是否属于豁免账户
5. 综合以上因素给出最终判定

**境外证券投资评估逻辑**

输入参数：
- 开户地区（sec_country）
- 持有人类型（sec_holder_type）
- 是否中国税收居民（sec_is_china_resident）
- 投资产品类型（sec_product_types）

评估流程：
1. 判断开户地区是否为CRS参与国
2. 判断持有人类型是否需要穿透（公司/信托持有）
3. 判断实际控制人的税收居民身份
4. 根据投资产品类型给出特别提示（私募基金、衍生品）
5. 综合判定风险等级

**离岸公司评估逻辑**

输入参数：
- 注册地区（company_country）
- 是否金融机构（company_is_fi）
- 经营性质（company_business_nature）
- 主动经营业务比例（company_active_ratio）
- 股权比例及控制权（company_ownership）
- 是否中国税收居民（company_is_china_resident）
- 其他控制人（company_other_controllers）
- 是否开立境外账户（company_has_foreign_account）

评估流程：
1. 判断注册地区是否为CRS参与国
2. 判断公司是否为金融机构
3. 如果不是金融机构，判断NFE分类（积极/消极）
4. 判断实际控制人及其税收居民身份
5. 判断是否开立境外金融账户
6. 综合判定是否需要穿透及风险等级

**境外保单评估逻辑**

输入参数：
- 保单性质（insurance_has_cash_value）
- 持有人（insurance_holder）
- 持有人税收居民身份（insurance_holder_resident）
- 被保险人（insurance_insured）
- 受益人（insurance_beneficiary）
- 受益人税收居民身份（insurance_beneficiary_resident）

评估流程：
1. 判断保单是否有现金价值（纯消费型直接判定无披露义务）
2. 判断持有人的税收居民身份
3. 判断受益人的税收居民身份（如已指定）
4. 检查持有人与被保险人是否分离
5. 综合判定风险等级

**离岸信托评估逻辑**

输入参数：
- 用户角色（trust_roles）
- 是否中国税收居民（trust_is_china_resident）
- 注入资产类型（trust_assets，委托人）
- 是否保留权力（trust_retained_power，委托人）
- 受益人类型（trust_beneficiary_type，受益人）
- 是否定期分配（trust_beneficiary_distribution，受益人）
- 是否有控制权（trust_protector_power，保护人）
- 信托活动类型（trust_activity_type）
- 是否开立境外账户（trust_has_foreign_account）

评估流程：
1. 根据信托活动类型判断CRS分类
2. 针对委托人角色：评估保留权力的影响
3. 针对受益人角色：评估固定受益人vs酌定受益人
4. 针对保护人角色：评估控制权的影响
5. 判断是否开立境外金融账户
6. 综合以上因素，按角色给出分类判定和风险等级

### 6.4 关键风险点识别

在完成各资产评估后，需要汇总识别以下关键风险点：

**风险点一：高净值账户未申报**

触发条件：境外银行存款余额超过250万美元 + 中国税收居民

**风险点二：消极NFE穿透**

触发条件：离岸公司属于消极NFE + 开立境外账户 + 控制人是中国税收居民

**风险点三：信托控制权保留**

触发条件：委托人保留撤销/更改受益人等权力 + 中国税收居民

**风险点四：未申报境外所得**

触发条件：境外证券投资有股息/利息收入 + 中国税收居民

**风险点五：保单分配未纳税**

触发条件：境外保单有分配 + 中国税收居民

## 七、报告模板设计

Skill需要提供清晰的报告模板，指导大模型生成结构化的诊断报告。

### 7.1 报告结构

报告采用结构化Markdown格式，包含六个主要部分：

**第一部分：诊断概要**

- 账套名称
- 诊断时间
- 涉及资产类型

**第二部分：资产总览**

以列表形式展示各资产类型是否涉及及风险等级

**第三部分：各资产信息披露义务判定**

分资产类型详细展示：
- 基本信息（用户回答汇总）
- CRS分类判定（如适用）
- CRS信息披露义务判定
- 信息交换预期
- 合规建议/特别提示

**第四部分：CRS信息交换风险评估**

- 总体风险等级
- 关键风险点列举

**第五部分：合规建议**

根据总体风险等级分级给出建议：
- 立即行动（1个月内）
- 短期行动（3-6个月内）
- 长期规划（6-12个月）

**第六部分：参考依据**

固定内容：CRS规则、中国税法相关规定

### 7.2 报告内容生成规则

**规则一：未涉及资产的处理**

对于用户未选择的资产类型，在对应章节显示"未涉及此类资产"

**规则二：动态内容生成**

根据用户回答和评估结果动态生成：
- 判定理由：结合具体回答内容说明判定逻辑
- 信息交换预期：根据开户地区说明信息交换时间表
- 合规建议：根据风险等级和具体情况给出针对性建议

**规则三：专业术语解释**

在使用专业术语时附加通俗解释，确保用户理解

**规则四：风险标识**

使用统一的风险标识：
- 高风险：🔴
- 中风险：🟡
- 低风险：🟢
- 有披露义务：✅
- 可能有披露义务：⚠️
- 无披露义务：❌

### 7.3 报告输出要求

**专业准确**

基于CRS规则和中国税法准确判定，不得给出错误的法律建议

**通俗易懂**

避免过多专业术语堆砌，用通俗语言解释复杂概念

**结构清晰**

严格遵循模板结构，使用标题、列表等元素增强可读性

**可操作性**

合规建议要具体可行，有明确的时间表和行动清单

**风险提示充分**

对高风险情况明确标识，避免误导用户低估风险

## 八、交互设计

Skill需要定义友好的交互文案，指导大模型如何与用户沟通。

### 8.1 问答交互规范

**问题展示格式**

每个问题展示时包含以下元素：
1. 进度提示：当前在哪个资产分类的第几个问题
2. 问题内容：核心问题文本
3. 问题说明：简要说明为什么问这个问题
4. 回答提示：引导用户如何回答
5. 选项列表（如果是选择题）

**用户输入提示**

- 单选题：提示"请选择一个选项"
- 多选题：提示"请选择所有适用的选项，可多选"
- 文本输入：提示"请输入您的回答"

**回答确认**

每个回答被记录前，向用户确认"您的回答是：xxx，是否正确？"

### 8.2 异常情况处理

**情况一：用户长时间未回答**

超过合理时间未收到回答时，发送温和提醒

**情况二：用户多次给出无效回答**

连续3次给出无效回答时：
1. 提供更详细的回答指导
2. 提供"跳过此问题"的选项
3. 标记该问题为"待确认"

**情况三：用户中途退出**

- 自动保存当前进度
- 提示用户可以随时继续

### 8.3 友好提示文案

**开始问答时**

"接下来我将通过一系列问题了解您的境外资产情况。整个过程大约需要10-15分钟，请您耐心配合。每个问题我会逐一询问，您只需要根据实际情况如实回答即可。"

**进入新资产分类时**

"好的，现在我们来了解您的{资产类型}相关情况。"

**完成一个资产分类时**

"关于{资产类型}的问题已经全部完成，接下来我们了解{下一个资产类型}。"

**用户回答无关内容时**

"抱歉，我没有理解您的回答。我们目前正在进行CRS信息披露诊断，当前问题是：{问题内容}。您可以{回答指导}。"

**用户回答不完整时**

"您的回答中提到了{已回答部分}，但还需要补充{缺失部分}的信息。请问{具体追问}？"

**全部问答完成时**

"太好了！所有问题都已完成。我正在根据您提供的信息生成诊断报告，请稍候..."

## 九、Skill文件结构设计

### 9.1 目录结构

Skill采用以下目录结构：

```
crs-diagnosis/
├── SKILL.md           # 主文件：角色定义、核心流程、指令
├── questions.md       # 完整问题清单及跳转逻辑
├── rules.md           # 评估规则和判定逻辑
└── templates.md       # 报告模板
```

### 9.2 SKILL.md 内容设计

SKILL.md是主文件，应控制在500行以内，包含：

**YAML元数据**

- name：技能名称，如"crs-diagnosis"
- description：触发场景描述，包含关键词

**角色定义**

定义大模型在此Skill中扮演的角色和能力

**核心原则**

强调逐题引导、答案完整性检查等关键原则

**工作流程**

定义三个阶段（问答、评估、报告）的概要流程

**辅助文件引用**

说明何时读取questions.md、rules.md、templates.md

**常见概念解释**

提供CRS、税收居民、消极NFE等术语的通俗解释

### 9.3 questions.md 内容设计

完整问题清单文件，包含：

**每个问题的结构**

- 问题ID
- 问题内容
- 问题说明
- 问题类型（单选/多选/文本）
- 选项列表（如适用）
- 跳转逻辑（下一个问题/条件跳转）
- 多要素标记（如适用）

### 9.4 rules.md 内容设计

评估规则文件，包含：

**CRS参与国名单**

列出主要CRS参与国家/地区，标注美国不参与

**实体分类规则**

积极NFE/消极NFE/投资实体的判定条件

**风险等级定义**

高/中/低风险的触发条件

**各资产评估逻辑**

每种资产类型的评估流程和判定条件

### 9.5 templates.md 内容设计

报告模板文件，包含：

**报告整体结构**

六部分的标准结构

**各部分模板**

每个部分的具体格式和占位符

**动态内容生成规则**

判定理由、合规建议等的生成逻辑

## 十、验证场景

用户在回答过程中询问"什么是税收居民"，验证Skill是否能解释后继续
